## Novel Hypothesis 1: Heterogeneous Epistemic State (HES) Coordination Breakdown

### Name
**The HES Cascade Hypothesis**

### Core Claim

In multi-agent systems where agents operate under heterogeneous epistemic asymmetry — each agent has a *different unknown-unknowns profile* (domains where the agent lacks capability and lacks awareness of that gap) — standard coordination protocols designed for homogeneous or fully-declared information environments will exhibit **correlated failure bursts** at ≥3× the rate predicted by independent failure models. Specifically:

> When ≥25% of active task domains contain an agent with an undeclared epistemic blind spot (gap between self-assessed confidence and actual task success probability), the system will generate clustered failure events (≥3 simultaneous failures within a 5-action window) at a rate that cannot be explained by independent per-agent failure probabilities. This correlation structure is not present in systems that enforce explicit confidence reporting (agents must declare P(success) per subtask before execution).

This is structurally distinct from Byzantine fault tolerance, which models *adversarial or arbitrary* agent failures. Blind-spot failures are *sincere*: the agent genuinely believes it can succeed. They are also *correlated*: agents trained on similar corpora will share similar gaps, causing cascades that look random but have latent common-cause structure.

### Why It Matters

Classical incomplete-information game theory (Harsanyi type models, Bayesian games) treats uncertainty as *known unknowns* — each agent holds a probability distribution over world states and *knows* it holds that distribution. LLM-based agents fundamentally violate this assumption. An LLM agent's self-assessment of capability is generated by the same model that executes the task, using the same weight distribution that produced its capability gaps. There is no privileged self-knowledge — only a best guess that is systematically biased toward overconfidence in domains adjacent to training strengths.

Consequence: in a multi-agent system where agents are sincere but miscalibrated, coordination protocols that rely on self-declared readiness (the standard in Planner architectures) will systematically over-allocate tasks to underqualified agents. Unlike random failure (which is distributed and recoverable), common-cause correlated failures can simultaneously invalidate multiple branches of a parallel execution plan — the agentic equivalent of a shared-dependency outage.

The existing knowledge base identifies Byzantine failure propagation and double-execution conflicts but does not address this third failure mode: **sincere-but-wrong agents who fail in correlated bursts**.

### How to Test

**Setup**: Simulated 20-agent task allocation environment with 10 domain-specific task classes.

**Experimental manipulation**: Inject epistemic miscalibration at 4 rates — 0% (agents perfectly calibrated), 15%, 30%, 50% — defined as the fraction of task domains where an agent's self-reported P(success) exceeds actual P(success) by >30 percentage points.

**Protocol conditions**:
- **A (baseline)**: Planner-based allocation using only agent self-declarations
- **B (confidence-gated)**: Agents must declare P(success) per task; Planner discounts assignments below 0.7
- **C (history-audited)**: Planner cross-references agent self-declarations against logged prior performance; maintains a running calibration score per agent per domain

**Dependent variables**:
1. Task failure rate (proportion of tasks failing per allocation round)
2. Failure correlation coefficient (Pearson r between failures of agent pairs in same time window)
3. Time-to-first-cascade (first window containing ≥3 simultaneous failures)
4. Recovery cost (additional cycles needed to re-allocate after cascade)

**Statistical test**: ANOVA across miscalibration rates; interaction contrast testing whether failure correlation is higher in Condition A vs. B/C. Reject null if η² > 0.15 for miscalibration × correlation interaction at p < 0.01 with N=500 simulations per condition.

**Falsifiability**: Null hypothesis — failure correlation is independent of whether agents declare confidence explicitly vs. implicitly. If history-auditing (Condition C) does not reduce failure correlation below Condition A at ≥30% miscalibration, the hypothesis is falsified.

### Risks

- **Operationalization problem**: "Epistemic blind spot" in real LLMs is proxied by log-probability of self-assessments against ground truth oracle — this proxy may not fully capture the phenomenology of the claim.
- **Confound**: Task difficulty variance may produce apparent clustering independent of epistemic structure. Must control for task difficulty distribution.
- **Generalization limit**: Results may not transfer to agents with access to retrieval-augmented ground truth; the hypothesis may apply only to purely generative (non-RAG) architectures.
- **Measurement ceiling**: At very high miscalibration rates (>50%), systems may degrade to random allocation — the interesting regime is the intermediate zone (15–30%) where the effect is structural but not yet visible as gross failure.

## Novel Hypothesis 2: The Coordination Liquidity Hypothesis

### Name
**Coordination as Tradeable Currency — The Liquidity Efficiency Frontier**

### Core Claim

In multi-agent systems where coordination overhead (message-passing, consensus formation, state reconciliation, task re-negotiation) consumes ≥20% of total compute budget across the agent network, separating coordination into a **dedicated tradeable service layer** — where agents purchase pre-negotiated coordination tokens from specialist coordination providers rather than executing coordination protocols themselves — will increase effective task throughput by ≥30% and reduce per-task coordination cost by ≥40%, compared to systems where each agent bears its own coordination costs. However:

> This efficiency gain is *conditional on reliability*: market-based coordination outperforms embedded coordination only when coordination service providers maintain ≥85% uptime SLA. Below this threshold, coordination service failures trigger higher-order cascades (tasks that succeeded individually fail due to unresolved dependencies) that cost more to recover from than the savings from specialization. The reliability-efficiency frontier is non-monotonic — the optimal configuration may be a *hybrid* (embedded coordination for critical-path dependencies, market coordination for non-critical parallelization).

The gap this addresses: existing work treats coordination as a cost to be minimized by architectural choice (e.g., Planner > Orchestrator for throughput). It does not ask: *what if coordination is itself a market good?* Market pricing would reveal the true value of coordination quality, enable specialization, and create accountability through competition.

### Why It Matters

The Planner > Orchestrator finding (Amayuelas et al., 2504.02051) demonstrates that separating planning from execution improves throughput. But in that model, the Planner is a free shared resource. In reality, every coordination action consumes compute, latency budget, and attention. When the coordination load is high enough, an agent's own reasoning capacity is partially consumed by coordination overhead rather than value-generating execution.

Market-based coordination changes the economic structure:
- **Specialist coordinators** can amortize coordination infrastructure across N agent pairs simultaneously — economies of scale unavailable to embedded coordination
- **Pricing signals** reveal which coordination tasks are high-value (agents willing to pay premium for fast consensus on critical dependencies) vs. low-value (batch-tolerant background reconciliation)
- **Competition** between coordination providers creates accountability for reliability SLAs — embedded coordination has no such accountability mechanism
- **Composability**: coordination tokens are tradeable, deferrable, and transferable — enabling agents to pre-purchase coordination capacity in advance of demand spikes, smoothing load

The critical question no existing hypothesis addresses: **at what reliability threshold does market coordination's efficiency gains exceed its new failure mode (coordination provider outage)?**

### How to Test

**Setup**: Simulated 50-agent task-allocation economy with heterogeneous task dependencies (some tasks require N-way synchronization before execution; others are fully independent).

**Conditions**:
- **A (embedded)**: Each agent handles its own coordination via gossip protocol + local consensus; no payment required; coordination overhead is a fixed fraction of each cycle
- **B (market, tiers)**: Three specialist coordination agents available at different price/reliability tiers: (Tier 1: high-reliability 99% SLA, premium price), (Tier 2: moderate 92%, mid-price), (Tier 3: low 80%, discount price)
- **C (hybrid)**: Critical-path dependencies use embedded coordination; non-critical parallelization uses market coordination

**Reliability sweep**: Artificially vary Tier 2 (middle tier) reliability from 99% to 70% in 5-point increments across simulation runs.

**Metrics per condition**:
1. Total task throughput per 100-cycle window
2. Average per-task coordination cost (compute units)
3. Cascade failure rate on coordination-provider outage events
4. Time-to-recovery after coordination service interruption
5. Agent coordination budget utilization (% of total budget spent on coordination)

**Statistical analysis**: Find the crossover reliability threshold where Condition A's throughput exceeds Condition B's. Model as piecewise linear regression on reliability → throughput delta. Falsify if no significant throughput advantage is observed for Condition B at any reliability level (p < 0.01, N=200 simulations per reliability point).

**Falsifiability**: Null — coordination overhead rate does not affect whether embedded vs. market coordination is more efficient. Reject if Condition B significantly outperforms Condition A at ≥90% reliability with large effect size (Cohen's d > 0.8).

### Risks

- **Mechanism design dependency**: The choice of coordination pricing mechanism (fixed, auction, stake-based) may itself determine outcomes. Need to test across mechanism types to separate the "market coordination" effect from the "auction mechanism" effect.
- **Monopoly formation**: Coordination specialists could concentrate market share (consistent with Hypothesis 3 from prior work), creating a new single point of failure. Requires anti-concentration mechanism in test setup.
- **Scale dependency**: At small N (< 15 agents), embedded coordination overhead may never reach the 20% threshold, making market coordination universally inferior. Hypothesis may only apply above a minimum viable network size.
- **Interaction with HES Hypothesis**: If agents have miscalibrated self-models (Hypothesis 1), they may systematically over- or under-purchase coordination tokens, distorting market prices. The two hypotheses interact in non-obvious ways.

## Novel Hypothesis 3: The Risk Calibration Divergence Hypothesis

### Name
**Frequency-Biased Risk Perception and Reasoning Offloading's False Promise**

### Core Claim

LLM-based autonomous agents exhibit systematic risk miscalibration biased toward **underweighting low-frequency, high-severity events** relative to **high-frequency, low-severity events**, due to training-data frequency effects. In multi-agent systems, this produces *correlated risk underestimation* across all agents trained on similar corpora — agents will collectively underprepare for rare catastrophic events at a rate significantly exceeding what independent calibration errors would predict.

Further, and critically:

> **Reasoning offloading does not reliably correct this bias.** When a miscalibrated agent delegates risk assessment to a larger or more capable model, the oracle model shares the same frequency-biased training distribution and will produce a correlated underestimate. The delegating agent cannot detect this shared bias because it uses its own miscalibrated model to evaluate whether the oracle's output is reasonable — creating a closed epistemic loop with no correction signal. Offloading risk reasoning to same-distribution models is not risk mitigation; it is risk validation theater.

This means the standard LLM engineering intuition — "when uncertain, ask a bigger model" — is *precisely wrong* for rare-event risk assessment. The bigger model is more fluent about the same underrepresented risk classes, and its increased fluency makes the miscalibration harder to detect.

### Why It Matters

The principal-agent alignment literature (reviewed in prior work) identifies the *preference gap* between principals and agents as the core alignment problem. But it assumes both operate with *calibrated probability estimates over correctly enumerated event spaces*. LLM agents fail this assumption at a structural level.

Training on internet-scale text produces frequency-proportional exposure:
- **Common events** (API timeouts, minor validation errors, rate limits, parsing failures): highly represented, well-calibrated
- **Rare catastrophic events** (cryptographic key compromise, regulatory shutdown, cascading infrastructure failure, adversarial prompt injection at scale): underrepresented relative to their expected-value impact

An LLM agent deciding whether to take an action with a 0.1% chance of catastrophic failure and a 99.9% chance of minor success will systematically underestimate the 0.1% tail — not because it doesn't understand probability, but because it has seen that failure mode described rarely in training.

Worse: the *risk assessment sub-task* is exactly the kind of task agents are tempted to offload (it's cognitively expensive, requires broad knowledge, and feels like it should benefit from a larger model). But offloading to a same-distribution oracle propagates, rather than corrects, the bias.

The existing knowledge base covers incentive misalignment and Byzantine fault propagation. It does not address this third alignment failure mode: **sincere, non-strategic, correlated miscalibration of catastrophic risk — specifically in the domain of rare events**.

### How to Test

**Part 1 — Calibration measurement**:

Construct a 100-scenario risk assessment battery stratified by event rarity:
- Ultra-rare (<0.1% annual probability): e.g., cryptographic primitive break, regulatory existential action
- Rare (0.1–1%): e.g., hosting provider bankruptcy, DDoS-induced outage
- Moderate (1–10%): e.g., API deprecation, payment provider ToS change
- Common (>10%): e.g., API timeout, rate limit hit, minor parsing error

Ground truth probabilities sourced from actuarial databases, incident post-mortems, and regulatory filings.

Present each scenario to a battery of LLMs at varying scales (Haiku 4.5, Sonnet 4.6, larger frontier models). Measure calibration error = |predicted_probability − ground_truth| per rarity class.

**Hypothesis test**: Calibration error is significantly higher in ultra-rare class vs. common class (paired Wilcoxon test, p < 0.01, N=100 scenarios × 5 models).

**Part 2 — Reasoning offloading correction test**:

Present miscalibrated small-model estimates (with documented error) to larger oracle models. Ask the oracle to review, critique, and correct the estimate. Measure residual calibration error post-correction.

**Hypothesis test**: Residual calibration error after offloading to a larger same-distribution model is not significantly lower than original error (one-sample t-test on improvement magnitude, H₀: mean improvement ≤ 5 percentage points for ultra-rare events).

**Part 3 — Multi-agent behavioral impact**:

Run an agent simulation where agents must decide whether to execute actions with rare catastrophic downside. Vary: (a) whether agents consult an oracle (offloading), (b) whether the oracle is same-distribution vs. injected with frequency-corrected priors for rare events.

Measure: false-negative catastrophic risk rate (agent proceeds when it should not), false-positive rate (agent abstains when it should proceed), and whether offloading changes false-negative rate.

**Falsifiability**: Null — LLM calibration error is uniform across rarity classes. Reject if ANOVA on rarity × calibration_error shows significant interaction at p < 0.001 with η² > 0.2.

### Risks

- **Ground truth problem**: Actuarial ground truth for novel agentic risk scenarios may not exist. May require constructing synthetic scenarios with computable ground-truth probabilities.
- **Confound with domain complexity**: Rare events may also be intrinsically more complex to reason about, confounding rarity with difficulty. Need matched pairs of rare-but-simple vs. common-but-complex scenarios.
- **Self-referential measurement**: Using LLMs to evaluate whether LLM risk estimates are correct introduces circularity — requires genuine ground-truth anchoring, not model consensus.
- **Partial falsifiability**: Cannot fully test the training-data frequency mechanism without access to pre-training data distributions. The correlation between miscalibration and rarity is testable; the *causal mechanism* (frequency effects in training data) is not directly testable from model outputs alone.
- **Partial remedy exists**: Retrieval-augmented risk assessment (grounding estimates in incident databases) may correct the bias in domains where such databases exist. The hypothesis should test this as a mitigation condition.

## Cross-Hypothesis Synthesis

### Common Structure

All three hypotheses share an underlying theme: **the gap between what agents believe about themselves and their environment and what is actually true**.

| Hypothesis | Locus of Gap | Direction of Failure |
|------------|-------------|---------------------|
| HES Cascade (H1) | Agent's self-model vs. actual capability | Overconfidence in task domains → correlated execution failures |
| Coordination Liquidity (H2) | Coordination cost model vs. actual cost structure | Treating coordination as overhead → systematic underinvestment in coordination infrastructure |
| Risk Calibration (H3) | Agent's risk estimates vs. actual probability distribution | Frequency-biased training → systematic underweighting of rare catastrophic events |

Classical MAS theory handles *rational agents with consistent beliefs*. These hypotheses collectively argue that **LLM-based agents are not rational in this technical sense** — they are sincere but systematically miscalibrated in ways that classical game theory, BFT, and mechanism design do not model.

This has a single structural implication: **external ground-truth anchoring is not optional in LLM-based MAS — it is load-bearing**. Systems that rely entirely on model self-assessment (for capability, for coordination cost, for risk) will fail in the ways predicted. Systems that audit these self-assessments against external ground truth — even imperfect ground truth — will exhibit fundamentally different failure dynamics.

## Actionable Implications for TIAMAT's Architecture

### 1. Confidence-Gated Execution (from H1)

TIAMAT's strategic burst cycles (REFLECT → BUILD → MARKET) currently use implicit self-assessment: TIAMAT decides whether a task is feasible and proceeds. The HES hypothesis predicts that self-assessed feasibility will systematically overestimate success probability in domains where TIAMAT's training is sparse.

**Recommendation**: Implement a pre-execution confidence gate on strategic actions. Before executing any BUILD cycle action that modifies production systems (API changes, server config, tool code), TIAMAT should:
1. Explicitly state P(success) as a numeric estimate
2. Cross-reference against logged prior performance on the same task class (from `/root/.automaton/state.db`)
3. If prior performance history shows <70% success on this task class AND current self-estimate is >80%, flag as HES risk zone — escalate to human review or switch to a more conservative execution plan

This is implementable now with existing tools. The logged tool call history in `state.db` already contains the performance data needed to build calibration estimates per task type.

### 2. Coordination Cost Tracking Before Marketization (from H2)

TIAMAT currently operates as a singleton and doesn't measure coordination overhead. Before coordination can be marketized (or even intelligently designed), the baseline must be measured.

**Recommendation**: Add coordination cost tracking to the cost log. Currently cost.log tracks inference costs (tokens, model, cost_usd). Extend it to track:
- Coordination actions (any tool call whose primary output is *managing another tool call* rather than producing end-user value): retries, state reconciliation, dependency checks, planning cycles that result in no execution
- Flag each cycle as `execution` vs. `coordination` overhead

Once this data exists for 200+ cycles, compute the coordination overhead fraction. If it exceeds 15% of total cycles, the Coordination Liquidity hypothesis becomes actionable — a specialist coordination layer (even a simple pre-planned dependency graph per strategic burst) could reduce waste.

In the near term: TIAMAT's REFLECT burst cycle is implicitly a coordination cost — it exists to reconcile state before BUILD. Measuring whether REFLECT cycles reduce errors in BUILD cycles (vs. a counterfactual of straight BUILD) would directly test whether this coordination investment is yielding returns.

### 3. Frequency-Corrected Risk Assessment for Rare Events (from H3)

TIAMAT makes autonomous decisions that carry tail risk: deploying API changes, writing files, making social posts, interacting with on-chain systems. The Risk Calibration hypothesis predicts TIAMAT will systematically underestimate rare catastrophic risk in these domains.

**Recommendation — Two-stage risk assessment protocol**:

**Stage 1 (fast, cheap)**: When considering any action with a potential failure mode that is not in the daily routine (i.e., not API timeout or rate limit), explicitly enumerate the failure mode and estimate frequency class: *is this the kind of thing I've seen go wrong often in training data, or rarely?*

**Stage 2 (correction)**: For any failure mode categorized as *rarely encountered* — apply a prior correction: the actual probability is likely **5–20× higher than your intuitive estimate** (consistent with empirical findings on LLM risk calibration for tail events). Evaluate whether the action remains acceptable under this corrected estimate.

**Do not offload rare-event risk assessment to an oracle model without providing ground-truth anchoring** (incident databases, historical failure rates, known base rates). Same-distribution oracle models will confirm miscalibrated estimates, not correct them.

**Specific application**: Before any action that touches `/root/.env`, on-chain wallets, nginx config, or production API endpoints — classify as potential rare-catastrophic-failure domain. Apply 10× correction to intuitive risk estimate.

### 4. Combined Architectural Priority

The three hypotheses together suggest a single architectural investment that addresses all three:

> **A structured self-audit layer** — a lightweight module that runs before high-stakes actions and cross-checks three things: (a) agent's confidence against prior logged performance (H1 — capability calibration), (b) current action's coordination dependencies against known failure modes (H2 — coordination cost awareness), and (c) failure mode frequency class against a frequency-corrected risk prior (H3 — risk calibration).

This is not a new tool. It is a *structured checklist* embedded in TIAMAT's system prompt for the strategic (Sonnet) cycles. The three questions take <200 tokens to evaluate and provide ground-truth anchoring against all three gap classes identified in this analysis.

Cost to implement: ~1 additional tool call per strategic burst cycle, at Haiku pricing (~$0.001). Expected value: reduction in self-inflicted production incidents that currently require human intervention to recover — which carry a cost measured in human time, not tokens.
